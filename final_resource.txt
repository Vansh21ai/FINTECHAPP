PROBLEM 1 — INVESTING FEAR  (Feature Name: "Risk Simulation Sandbox")
--------------------------------------------------------------------------------

WHAT THE PROBLEM IS:
  Young users in India want to invest money but are too scared of losing it.
  They do not understand how markets work, what mutual funds are, or what
  "portfolio" means. So they do nothing and just let money sit idle.

WHAT OUR APP SOLVES:
  We remove the fear by letting users practice with fake money first.
  No real money is ever at risk during simulation. Once confident, they invest real.

HOW IT WORKS INSIDE THE APP:
  - User opens the "Sandbox" tab on the dashboard.
  - They enter an amount (fake money) like Rs. 10,000.
  - They pick an investment type: Mutual Fund / Stocks / Gold.
  - The app fast-forwards time using historical market data.
  - The graph shows what would have happened to their money in real past events
    (e.g., COVID crash 2020, post-COVID recovery 2021).
  - An AI explainer box beside the graph translates complex events into
    simple sentences: "Your money dropped here because markets panicked.
    But it recovered in 8 months."

GRAPH REQUIRED?  YES
  - An interactive line graph (like a stock chart) that goes UP and DOWN
    based on historical data simulation.
  - A "Loss Probability Meter" — a speedometer-style circular gauge.
    Green = Low Risk. Yellow = Medium. Red = High Risk.
  - Both charts update in real-time as the user changes their input amount.

TOOL TO BUILD IT:
  ApexCharts (free) — supports interactive line charts and radial gauges.

PROBLEM 2 — TRUST AND TRANSPARENCY DEFICIT  (Feature: "Consent Ledger")
--------------------------------------------------------------------------------

WHAT THE PROBLEM IS:
  Gen-Z does not trust financial apps. They believe their data is being sold.
  They hate hidden fees. They do not understand why a payment got blocked.
  73% of Gen-Z still prefer banks over apps purely because of trust.

WHAT OUR APP SOLVES:
  We make every data access and every transaction decision visible to the user.
  Nothing is hidden. The user can see exactly what the app did with their data
  and why every financial action happened.

HOW IT WORKS INSIDE THE APP:

  PART A — THE CONSENT LEDGER (Most Powerful Trust Feature):
  Every single time our app reads user data, it writes one visible log line
  that the user can open anytime. Example of what they see:

      [10:32 AM]  Location accessed       → Reason: Fraud check on payment
      [10:33 AM]  Spending history read   → Reason: Building investment tip for you
      [10:35 AM]  Phone model checked     → Reason: Checking gadget insurance fit

  This is technically just an append-only log in PostgreSQL.
  But for the user, it is the most transparent experience in any Indian fintech app.
  NO competitor does this. This is a real differentiator.

  PART B — EXPLAINABLE TRANSACTION ENGINE:
  If a payment fails or gets flagged, instead of a generic error message,
  the app gives a human-friendly reason:
      "Payment stopped — this merchant was reported 5 times this week."
      "Transaction flagged — unusual location. Was this you?"

GRAPH REQUIRED?  YES (simple visual)
  - A "Trust Score" circular progress bar (0 to 100).
    Shows how secure the account is based on MFA, KYC, login history.
  - A color-coded live scrolling feed for the Consent Ledger.
    Green = safe data access. Yellow = advisory. Red = sensitive access.

TOOL TO BUILD IT:
  ApexCharts (radial bar for trust score). Live feed built with React state updates.

PROBLEM 3 — CASHBACK DEPENDENCY  (Feature: "Micro-Invest + Financial XP System")
--------------------------------------------------------------------------------

WHAT THE PROBLEM IS:
  Every financial app gives Rs. 5 cashback to trick users into spending more.
  This makes users spend unnecessarily just to get a small reward.
  It builds zero wealth and creates a bad financial habit loop.

WHAT OUR APP SOLVES:
  We replace cashback with "Wealth-back."
  Every reward the user earns is automatically invested into a micro mutual fund.
  On top of this, we add a Financial XP System (borrowed from gaming)
  to make saving money feel rewarding and fun.

HOW IT WORKS INSIDE THE APP:

  PART A — AUTO MICRO-INVEST:
  User buys a coffee for Rs. 120.
  App gives a reward of Rs. 6 (5% back).
  Instead of crediting Rs. 6 to wallet, it auto-invests into a liquid mutual fund.
  Over time, the user watches their "invested rewards" grow on the dashboard.

  PART B — FINANCIAL XP SYSTEM (NEW ADDITION):
  Users earn XP points for healthy financial actions:

      Action                             XP Earned
      ─────────────────────────────────────────────
      Complete KYC verification          +100 XP
      First micro-invest done            +200 XP
      Saved consistently for 3 months   +500 XP
      Ran a Risk Simulation              +150 XP
      Added an insurance cover           +300 XP

  Five Tiers:
      Beginner → Saver → Investor → Wealth Builder → Money Master

  When a user reaches "Investor" tier, the Risk Sandbox unlocks
  with real (not just historical) market data. This is a feature-gate mechanic
  used by real apps like Duolingo and Robinhood to drive engagement.

  Milestones trigger gamified UI moments:
      - Confetti animation when Rs. 1000 in rewards is saved.
      - Badge unlock animation using LottieFiles.
      - "Level UP!" card slides up from bottom of screen.

GRAPH REQUIRED?  YES
  - A "Compound Interest Curve" line graph.
    Shows the user: "If you keep saving this cashback for 2 years,
    your balance reaches this point." The curve rises dramatically to show
    the power of compounding even on small amounts.
  - A horizontal XP Progress Bar permanently visible in the dashboard header.
    Shows current tier and XP needed for next level. Always motivating.

TOOL TO BUILD IT:
  ApexCharts (area/line for compound curve). Lottie for badge animations.

PROBLEM 4 — INSURANCE COMPLEXITY  (Feature: "Coverage Visualizer + Scenario Sim")
--------------------------------------------------------------------------------

WHAT THE PROBLEM IS:
  Insurance policies are written in legal language that no 20-year-old will read.
  Youth skip insurance entirely because it feels confusing, boring, and irrelevant.
  They only realize they needed it after something goes wrong.

WHAT OUR APP SOLVES:
  We translate all insurance into plain English.
  Then we make it interactive — users click scenarios to see if they are protected.
  Insurance feels less like a legal contract and more like a personal safety tool.

HOW IT WORKS INSIDE THE APP:

  SCENARIO SIMULATOR:
  User sees a set of scenario buttons:
      "What if I drop my phone in a pool?"
      "What if my laptop gets stolen?"
      "What if I get in a bike accident?"
      "What if a cyber fraud empties my account?"

  User clicks one. App instantly shows:
      COVERED  ✅  — "Your Gadget Plan covers accidental water damage."
      NOT COVERED ❌ — "Your current plan does NOT cover theft above Rs. 30,000.
                        Upgrade for Rs. 29/month."

  This makes the abstract concept of risk feel immediate and personal.
  Embedded insurance: when a user buys electronics, the app automatically
  suggests gadget insurance at that exact moment (embedded at point of purchase).

GRAPH REQUIRED?  NOT a traditional line graph.
  - A 3D visual of an umbrella covering icons (phone, laptop, bike, wallet).
    Icons that are covered glow green. Icons NOT covered pulse red.
  - Built using Spline (free 3D tool) embedded into the React dashboard.
  - Toggle switches let users add coverage and watch icons turn from red to green
    in real time. Extremely visual. Extremely engaging for Gen-Z.

TOOL TO BUILD IT:
  Spline for 3D umbrella model. React toggle state for covered/not-covered logic.

SECTION 2 — THE ARCHITECTURAL HEART (MOST IMPORTANT NEW ADDITION)
================================================================================

This is what turns 4 separate features into ONE super app.
Without this, you have 4 disconnected tools, not a super app.

--------------------------------------------------------------------------------
THE FINANCIAL EVENT BUS
--------------------------------------------------------------------------------

WHY IT WAS ADDED:
  Every feature in the app needs to react to what the user does in real time.
  When a user buys a phone, multiple things must happen simultaneously:
  the insurance module should suggest gadget cover, the cashback module should
  auto-invest the reward, the trust ledger should log the transaction, and the
  investment module should note the user is interested in electronics.

  Without an Event Bus, you would have to manually connect each module to each
  other. That becomes a spaghetti mess. The Event Bus is the clean solution.

HOW IT WORKS:

  USER BUYS A PHONE (Rs. 45,000)
              ↓
       FINANCIAL EVENT BUS  (Node.js EventEmitter / BullMQ)
       ↙          ↓             ↓              ↘
  Insurance    Cashback       Trust          Investment
  Module       Module         Ledger         Sandbox
  auto-        converts       logs clean     suggests
  suggests     reward to      transaction    "phone sector
  gadget       micro-invest                  stocks" tip
  cover

  Each module listens for events independently.
  When an event fires, each module handles its own job.
  They do not interfere with each other.

TECHNICAL REQUIREMENT:
  Built using Node.js EventEmitter (zero cost, built-in) for hackathon.
  In production this would scale to BullMQ (free, Redis-backed queue).
  This is literally how Razorpay and CRED handle multi-module triggers internally.


--------------------------------------------------------------------------------
THE 3-QUESTION ONBOARDING FLOW (CRITICAL MISSING PIECE — NOW ADDED)
--------------------------------------------------------------------------------

WHY IT WAS ADDED:
  Gen-Z abandons any app that takes more than 30 seconds to understand.
  Without onboarding, every user sees the same dashboard regardless of their needs.
  The 3-question flow personalizes the entire experience from second one.

HOW IT WORKS:
  On first login, the app asks exactly 3 questions (no more, no less):

      QUESTION 1: "What is your monthly income range?"
          Options: Below Rs. 15,000  /  Rs. 15,000–40,000  /  Rs. 40,000+

      QUESTION 2: "What is your biggest financial worry right now?"
          Options: Losing money  /  Not saving enough  /  Don't understand investing

      QUESTION 3: "Pick your first financial goal"
          Options: Emergency fund  /  Travel fund  /  Start a business

  Based on the answers, the dashboard pre-configures itself:
      - User scared of losses → Trust Dashboard is shown prominently.
      - User wants to save → Micro-Invest tracker is front and center.
      - User wants to invest → Risk Sandbox is the first thing they see.

  This is a JSON config map. Takes 20 minutes to build.
  Impact on user retention is enormous. Every real fintech app does this.

GRAPH REQUIRED?  NO (just animated card stack UI for the 3 questions)
TOOL TO BUILD IT:
  React state + Framer Motion (free) for smooth card transition animations.

--------------------------------------------------------------------------------
THE APPEND-ONLY LEDGER (HOW MONEY IS TRACKED SAFELY)
--------------------------------------------------------------------------------

WHY IT IS USED:
  In a normal database, when a user spends money, the balance gets UPDATED.
  If something goes wrong, you cannot trace what happened. Money can disappear.
  The append-only ledger NEVER updates. Every transaction is a NEW row INSERT.
  The current balance is always calculated as the sum of all past entries.

  Current Balance = Sum of all transaction entries for that user

  If anything ever goes wrong, every single step is traceable.
  This is how real banks store money. It is mathematically auditable.

  For fast dashboard reads (so the user does not wait for a sum calculation
  every time), we store a pre-computed balance in Redis cache.
  Write side → PostgreSQL ledger. Read side → Redis cache.
  This pattern is called CQRS (Command Query Responsibility Segregation).

DEVELOPER 1 — THE ARCHITECT (Backend + Infrastructure)
================================================================================

YOUR JOB:
  You are building the brain of the entire app.
  Without your work, nothing saves, nothing moves, nothing is secure.

WHAT YOU WILL BUILD:

  1. Node.js + Express API Server
     → The central backend that all frontend calls hit.
     → Set up all routes: /invest, /rewards, /insurance, /trust-log, /user.

  2. PostgreSQL Database Schema (on Neon — free)
     → Create the append-only ledger table.
     → Create the consent log table for the Trust feature.
     → Create the user XP and tier table.
     → Create the transactions and rewards tables.

  3. Financial Event Bus
     → Set up BullMQ with Upstash Redis (both free).
     → Create 4 event queues: invest-event, reward-event, insurance-event, trust-event.
     → Wire up event producers (fired on each transaction).
     → Wire up event consumers (each module handles its own queue).

  4. JWT Authentication + Secure Onboarding API
     → User signs up → JWT token issued → stored in HttpOnly cookie.
     → Onboarding answers saved to DB → used to configure dashboard response.

  5. Security Protocols
     → bcrypt for password hashing.
     → Rate limiting on all financial endpoints (express-rate-limit — free).
     → Environment variables managed via .env file (never commit this).

  BRIEF EXPLANATION OF EACH PART:
     Node.js API    → Handles all communication between frontend and database.
     PostgreSQL      → Stores all financial data in an auditable, append-only format.
     Event Bus       → Makes all 4 modules react to the same user action simultaneously.
     JWT Auth        → Keeps the user session secure without storing passwords in plain text.
     Rate Limiting   → Prevents bots or attackers from hammering financial endpoints.

SECTION 7 — DEPLOYMENT (FREE + FAST — NO WAITING FOR USERS)
================================================================================

WHY VERCEL WAS REPLACED:
  Vercel's free tier suffers from cold starts on serverless functions.
  This means the first user to hit the app after a period of inactivity
  may wait 3-10 seconds for the backend to wake up.
  For a financial app demo, this is unacceptable.
  The stack below is faster, always-on, and completely free.


--------------------------------------------------------------------------------
FRONTEND DEPLOYMENT — CLOUDFLARE PAGES
--------------------------------------------------------------------------------
  WHY IT IS FASTER THAN VERCEL:
    Cloudflare Pages deploys to Cloudflare's global edge network (300+ cities).
    Your React app is served from a server physically close to every user.
    There are NO cold starts. The frontend is always instant.
    No sleeping. No waiting. Every page load is under 100ms.

  HOW TO DEPLOY:
    1. Push React app to GitHub.
    2. Go to https://pages.cloudflare.com
    3. Connect your GitHub repository.
    4. Set build command: npm run build
    5. Set output directory: dist
    6. Click Deploy. Done. You get a free .pages.dev URL.
    7. Every future git push auto-deploys in under 60 seconds.

  COST: Completely free. No credit card needed.


--------------------------------------------------------------------------------
BACKEND DEPLOYMENT — RAILWAY
--------------------------------------------------------------------------------
  WHY RAILWAY:
    Railway runs your Node.js server as a persistent, always-on process.
    There are NO cold starts. The backend never sleeps.
    Free tier gives enough hours for a hackathon with no issues.
    Deployment is as simple as connecting a GitHub repo.

  HOW TO DEPLOY:
    1. Go to https://railway.app
    2. Connect GitHub → select your backend repo folder.
    3. Railway auto-detects Node.js and runs npm start.
    4. Add your .env variables in the Railway dashboard (Environment tab).
    5. Get your Railway URL. Add it as the API base URL in the React frontend.

  COST: Free tier covers the hackathon duration fully.


--------------------------------------------------------------------------------
DATABASE — NEON (FREE SERVERLESS POSTGRESQL)
--------------------------------------------------------------------------------
  WHY NEON:
    Neon is a free serverless PostgreSQL service.
    It wakes up in milliseconds (much faster than Supabase free tier).
    It supports connection pooling out of the box.
    Free tier gives 0.5 GB storage — more than enough for a hackathon.

  HOW TO USE:
    1. Go to https://neon.tech and create a free project.
    2. Get your connection string.
    3. Add it to your .env as DATABASE_URL.
    4. Use the pg npm package in Node.js to connect.

  COST: Completely free.


--------------------------------------------------------------------------------
REDIS CACHE + QUEUE — UPSTASH
--------------------------------------------------------------------------------
  WHY UPSTASH:
    Upstash is a free serverless Redis service.
    Used for two things in our app:
      1. Caching user balances (CQRS read side — fast dashboard loads).
      2. Powering BullMQ job queues (Financial Event Bus).
    Free tier: 10,000 commands/day. More than sufficient for a demo.

  HOW TO USE:
    1. Go to https://upstash.com and create a free Redis database.
    2. Get the UPSTASH_REDIS_REST_URL and UPSTASH_REDIS_REST_TOKEN.
    3. Add to .env. BullMQ connects to it automatically.

  COST: Completely free.

SECTION 8 — ANDROID STUDIO (TWA WRAPPER — CLEAR EXPLANATION)
================================================================================

WHAT ANDROID STUDIO DOES IN THIS PROJECT:
  Android Studio does NOT rebuild the app.
  It wraps your already-deployed React web app into a native Android shell.
  The result is a full-screen Android app with NO browser address bar.
  It looks and behaves exactly like a native app during the demo.
  This technique is called Trusted Web Activity (TWA).
  Twitter Lite and real production companies use this exact approach.

WHO DOES THIS: Developer 2 (Frontend Engineer). Takes 3-4 hours.

STEP BY STEP:

  Step 1:  Make sure the React app is deployed on Cloudflare Pages first.
           You need a live URL before Android Studio can wrap it.

  Step 2:  Open Android Studio → New Project → Select "No Activity" template.
           This gives a clean Android shell with no extra code.

  Step 3:  Open the app-level build.gradle file.
           Add this dependency:
             dependencies {
               implementation 'com.google.androidbrowserhelper:androidbrowserhelper:2.5.0'
             }

  Step 4:  Open AndroidManifest.xml.
           Add the TWA Activity and point it to your Cloudflare Pages URL:
             <meta-data android:name="asset_statements" android:value="@string/asset_statements"/>
           Set the URL to your deployed frontend link.

  Step 5:  DIGITAL ASSET LINK — This is the most critical step.
           In Android Studio terminal, generate the SHA-256 fingerprint:
             keytool -list -v -keystore ~/.android/debug.keystore -alias androiddebugkey -storepass android -keypass android
           Copy the SHA-256 fingerprint.
           Create a file called assetlinks.json with this content:
             [{
               "relation": ["delegate_permission/common.handle_all_urls"],
               "target": {
                 "namespace": "android_app",
                 "package_name": "com.yourteam.finapp",
                 "sha256_cert_fingerprints": ["YOUR_SHA256_HERE"]
               }
             }]
           Host this file on your Cloudflare Pages domain at:
             https://yourdomain.pages.dev/.well-known/assetlinks.json
           This proves to Android that you own the web app being wrapped.
           Without this, the browser bar appears. With it, the bar disappears.

  Step 6:  Run on Android Emulator or a physical phone via USB.
           The app opens full screen. No browser UI visible.
           This is your demo-ready Android app.
